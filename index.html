מה זה ולמה צריך אותו

לקוחות Konimbo שמוכרים מוצרים צריכים לדעת בכל רגע נתון מה המחיר של המוצרים שלהם ביחס למתחרים — KSP, Bug, Ivory, ועוד. ניטור ידני של עשרות ומאות מוצרים בחנויות שונות הוא בלתי אפשרי.

פלואו ה-Spider פותר את זה אוטומטית: הוא מוציא מחירים עדכניים מחנויות מתחרות דרך שירות Spider-UI, מצלב אותם עם המוצרים שבחנות של הלקוח ב-Medusa, מחשב מחיר מומלץ למכירה, וכותב הכל ישירות ל-metadata של כל מוצר.

מוקדמות

שדהתיאורדוגמה / הערות

2. Tag ב-Medusa

צריך להתקיים tag בשם התואם ל-tag_name שבשורת Baserow. הפלואו עובד רק על מוצרים עם תג זה.

מדוע Tag ולא Collection: ב-Medusa כל מוצר יכול להשתייך לקולקציה אחת בלבד. שימוש ב-tag מאפשר ללקוח לתייג מוצר ב-spider ובמקביל לשייכו לקולקציה משלו ללא הגבלה.

3. Metadata על variants ב-Medusa

כל variant שרוצים לכלול בהשוואה צריך metadata תחת מפתח spider במבנה הבא:

{
  "spider": {
    "store_1": "ksp",
    "url_product_1": "https://ksp.co.il/web/cat/product/12345",
    "product_category_1": "tvs",
    "store_2": "bug",
    "url_product_2": "https://bug.co.il/p/67890",
    "product_category_2": "tvs",
    "calculated_expression": "{{ final_price | times: 0.9 }}"
  }
}

שמות המפתחות: store_N, url_product_N, product_category_N

calculated_expression הוא per-variant — כל מוצר יכול לקבל נוסחה שונה. אם לא קיים — ה-expression מ-Baserow ישמש כ-fallback; אם גם הוא לא קיים — מחיר ה-Spider עובר ללא שינוי.

רשימת חנויות נתמכות: נטענת דינמית מ-Spider API לפי הקטגוריות הקיימות במוצרים — אין רשימה hardcoded. variant עם חנות לא נתמכת מסומן אוטומטית ומופיע בדוח המייל.

תיאור הצמתים

1
get_shops

מה עושה: שולח GET ל-Baserow API ומחזיר רשימת חנויות.

API: https://table.konimbo.co.il/api/database/rows/table/2649/?user_field_names=true

פלט: Baserow response מלא; הפלואו משתמש ב-.results[] לאיטרציה.

הערה: ה-Baserow API token מוגדר hardcoded בקוד.

לולאה — parallel: false · skip_failures: true
הפלואו רץ על כל חנות ברצף. כשל בצומת כלשהו גורם לדילוג על החנות ללא התראה. ראה Issue 2.
2
get_tags

מה עושה: בודק אם החנות פעילה (status), ואם כן — מחפש tag ב-Medusa Admin לפי tag_name. מבצע exact match (case-insensitive, trim).

קלט: החנות הנוכחית מהאיטרטור (flow_input.iter.value) — ✅ wiring נכון.

API: {base_url}admin/product-tags?q={tag_name}&limit=100

פלט:

  • status לא פעיל: { skip: true, reason: "status_disabled" }
  • tag לא נמצא: { error: "...", skip: true }
  • הצלחה: { tag_id, tag_value, skip: false }

בשגיאה: מחזיר { skip: true } → הצומת הבא בודק את הדגל ומחזיר תוצאה ריקה בצורה מבוקרת.

3
get_data_from_shop

מה עושה: אם ה-tag תקין — מושך עד 1,000 מוצרים לפי tag_id. שולח קריאת getdestributors ל-Spider API לכל קטגוריה שנמצאת במוצרים, ובונה דינמית את רשימת החנויות הנתמכות. מסנן variants ומפריד בין נתמך ולא נתמך.

API (מוצרים): {base_url}admin/products?tag_id[]={tag_id}&limit=1000

API (vendors): https://spider-ui.rudenetworks.com/admin/v1/?method=getdestributors&category={category}&token=<API_TOKEN>

הגבלה: עד 1,000 מוצרים — אם התג יגדל יש צורך ב-pagination.

פלט: { total, by_vendor, variants: [...], unsupported: [...] }

variants לא נתמכים: מועברים ישירות ל-not_found עם סיבה "Store not supported by Spider: ..."

⚠️ Wiring Bug (ריבוי חנויות): לוקח תמיד store_data[0]. עובד עם חנות אחת; עם מספר חנויות — תמיד credentials של חנות 1. ראה Issue 1.
4
create_set

מה עושה: מקבץ variants לפי store|category — קבוצה אחת = קריאה אחת ל-Spider. מבצע deduplication לפי product_id + variant_id + url. מעביר unsupported הלאה ללא שינוי.

פלט: { groups: [{ store, category, data: [...] }], unsupported: [...] }

5
search_urls

מה עושה: לכל קבוצה שולח GET ל-Spider API ומצלב URLs (עם normalization: הסרת פרוטוקול + trailing slash + lowercase). מוסיף unsupported ישירות ל-not_found.

Spider API: https://spider-ui.rudenetworks.com/admin/v1/?method=resultdestributor&distributor={store}&category={category}&token=<API_TOKEN>

טיפול בשגיאת רשת: אם קריאה נכשלת, כל ה-variants של אותה קבוצה מתווספים ל-not_found עם price = -1 וסיבת השגיאה — לא נעלמים בשקט.

פלט: { found: [...], not_found: [...] }

not_found כולל: URL שגוי / לא נמצא · vendor לא נתמך · שגיאת רשת.

6
calculate_price

מה עושה: לכל variant ב-found — מוצא את המחיר הנמוך ביותר בין כל החנויות שנמצאו, ואז מחשב final_price על ידי הפעלת calculated_expression.

לוגיקת החישוב:

  1. קיבוץ לפי variant_id — כל שורות אותו variant מקובצות
  2. בחירת המחיר הנמוך ביותר (> 0) בין כל החנויות
  3. הפעלת calculated_expression מ-metadata.spider עם final_price כמשתנה
  4. הוספת final_price המחושב לכל שורות ה-variant

דוגמה: ביטוי {{ final_price | times: 0.9 }} + KSP=1,000₪, Bug=900₪ → final_price = 810₪

פלט: { total, processed, results: [...] } — המערך בתוך מפתח results.

7
post_medusa

מה עושה: כותב תוצאות ל-Medusa. כל הבקשות נשלחות במקביל (Promise.allSettled). המטאדאטה נכתב תחת מפתח spider ומשמר שדות קיימים.

מה נכתב ל-metadata (תחת spider):

שדהתוכן
store_Nשם החנות המתחרה
url_product_Nקישור המוצר בחנות המתחרה
product_category_Nקטגוריה ב-Spider
price_Nמחיר ב-Spider (-1 אם לא נמצא)
final_priceמחיר סופי לאחר חישוב expression (מ-calculate_price)

Payload: { "metadata": { "spider": { "store_1": "ksp", "url_product_1": "...", ... } } }

Scope: variant (ברירת מחדל): POST {base_url}admin/products/{pid}/variants/{vid}
product: POST {base_url}admin/products/{pid}

⚠️ Wiring Bug (ריבוי חנויות): גם post_medusa לוקח store_data[0]. ראה Issue 1.
8
sent_mail

מה עושה: שולח דוח HTML עם רשימת not_found — כולל URLים שגויים, vendors לא נתמכים, ושגיאות רשת.

מצב נוכחי: נשלח לאיתמר (itamarh@konimbo.co.il) לצורך בדיקה פנימית בלבד — לא חלק מהמוצר הסופי.

SMTP: Gmail + app password, hardcoded ב-flow.yaml.

עתידי: לקבוע מי מקבל בproduction (לקוח? צוות Konimbo? שניהם?).

Known Issues

אזהרה Issue 1 — Wiring שגוי עם ריבוי חנויות

get_data_from_shop ו-post_medusa מקבלים את כל המערך מ-Baserow ולוקחים תמיד [0].

  • חנות אחת כרגע: [0] = החנות היחידה = תקין
  • עם מספר חנויות: כל האיטרציות ישתמשו בcredentials של חנות 1 — גם כשמעבדים חנות 2 או 3

תיקון: לשנות store_data של שני הצמתים מ-results.j.results ל-flow_input.iter.value (כמו get_tags).

דחיפות: לתקן לפני הוספת חנות שנייה ל-Baserow.

אזהרה Issue 2 — אין התראה על כשלים בזמן ריצה

כשחנות נכשלת מסיבות runtime (token פג, Medusa לא זמינה), skip_failures: true מדלג עליה בשקט. אין מייל, אין לוג מרכזי — רק שגיאה אדומה בממשק Windmill.

שים לב: חנות עם status = false ב-Baserow נדלגת בכוונה עם לוג ברור — זה לא בעיה. הבעיה היא כשלים בלתי צפויים.

תיקון מוצע: לאסוף שמות חנויות שנכשלו ולהכניסן לדוח המייל של sent_mail.

ידוע Issue 3 — בעיית האימות (JOK Token)

הפלואו מאמת מול Medusa ב-Authorization: Bearer {JOK token}. הטוקן נשמר ב-Baserow.

  • Tokens עלולים לפוג תוקף ללא אזהרה מוקדמת
  • לא סקיילאבלי — לכל לקוח חדש צריך ליצור משתמש ב-Medusa ידנית
  • שמירת credentials ב-Baserow אינה פתרון מאובטח ל-production

הסיבה: אין מנגנון API Key ב-Medusa ל-Admin API. JOK הוא הדרך היחידה כרגע.

לחקור: תמיכה ב-API Keys בגרסאות עתידיות; אחסון ב-Windmill Secrets.
